## Browser-verify tips (learned the hard way)

- **Preview MCP screenshots are unreliable for sub-pixel detail.** JPEG compression smears 1-2px features (scanlines, dither stipple) into uniform tints; resize timing can produce thumbnail-sized images that look like rendering bugs. **If a screenshot contradicts intuition, sample canvas pixels via `getImageData` first** — they're more reliable than the screenshot tool for detecting real issues. Even better: ask the user to look in their native browser.
- **`preview_click` selector dispatch sometimes lands outside the clickable region** in a narrow preview viewport. When click "succeeds" but no event fires, fall back to `preview_eval` with `element.click()`.
- **The preview MCP duplicates console output 6×** for reasons not yet understood. Log de-duplication is purely cosmetic; the events are still firing once per emit.
- **Force-verifying defeat/victory paths** when natural runs keep winning/losing the wrong way: temporarily add `(window as unknown as { __bus: typeof this.bus }).__bus = this.bus;` to Game constructor, then `window.__bus.emit('battle:ended', { winner: 'enemy' })` or `window.__bus.emit('run:victory', {})` from eval. Remove the hook before committing.
- **`getImageData` on a WebGL canvas can return cleared buffers** if `preserveDrawingBuffer` is false (Three.js default). Read pixels synchronously inside the same frame the render happened, or temporarily set `preserveDrawingBuffer: true` for debugging.
- **Verifying short-duration sprite fades (D5.C-style) is hard via the preview MCP.** Each `preview_eval` synchronously blocks the JS main thread, stalling `requestAnimationFrame`. When the next frame fires, `dt` is the full eval+sleep time — easily 100-200ms per frame instead of 16ms — and the renderer's animator drains a 500ms fade in 2-3 frames. The fade IS correctly wired (you can prove it by inspecting `br.animator.fades.size` immediately after the spawn fires), it just visually completes faster than you can screenshot. To capture a clean mid-progress visual: trigger the fade, freeze `br.animator.update` to a no-op, then manually call `sprites.updateSprite(handle, { alpha: 0.5 })` to lock the alpha — that survives across screenshots and evals. Bumping `SPAWN.durationSeconds` to a longer value in `config/spawn.json` is the alternative (and works for any short fade) but requires a rebuild.
- **A backgrounded preview tab throttles `requestAnimationFrame` to ~0, so the Clock-driven sim freezes at `tick 0` and `preview_screenshot` times out (the WebGL render loop never goes idle for the capture).** This bit E6's eyeball verification hard. Two workarounds, both via the dev `window.__game` handle (DEV-only, `src/main.ts`): (1) **drive the loop by hand** — `const br = __game.activeScene.battleRenderer; for (let i=0;i<600 && !ended;i++){ br.world.tick(); br.update(1/20); }` advances sim + animations deterministically without rAF (TS `private` fields are runtime-accessible, so `br.world` / `br.animator` / `br.handles` / `br.projectiles` are all reachable). (2) **inspect synchronously** — subscribe to `__game.bus.on('unit:attacked', …)` and read DOM/positions in the handler; the BattleRenderer's handler runs first (subscription order) so the shove/hitsplat/projectile already exists when yours fires. The tab sometimes un-throttles mid-session (becomes active), at which point combat auto-runs to completion on the real loop — don't be surprised if `activeScene` has advanced to PromotionScene between evals; persist probes on `window.__x` so they survive the scene swap. Net: **functional correctness is fully eval-verifiable; subjective feel still needs the user's native browser.**
