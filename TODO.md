- [x] **#2 ‚Äî Billboard sprite depth sorting: a melee attacker can render IN FRONT of a target that's actually closer to the camera.** ‚úÖ **FIXED (Qb, 2026-06-18) via Option 1 (per-frame depth sort).** Confirmed cause (a): the glyph billboards are ONE instanced draw call with `depthWrite: false` ([SpriteRenderer](src/render/SpriteRenderer.ts):129 ‚Äî a transparent blend can't share the single-value depth buffer, so depth-write is off and the soft glyph edges don't clip neighbors), and there was **no per-instance depth sort** ‚Äî so sprite-vs-sprite occlusion fell to **draw order = instance slot order = spawn order**, and a farther unit in a higher slot painted over a nearer one. The **E6.A shove** (cause b) only AMPLIFIES it ‚Äî it lunges the attacker's quad camera-ward so the overlap shows exactly during a strike ‚Äî it's not the root (the user picked Option 1 after we ruled out alphaTest+depthWrite, which would've fought the alpha fades). **Fix:** `SpriteRenderer.sortByDepth(camera)` reorders all six per-instance attribute buffers + the handle‚áÑslot maps back-to-front (farthest ‚Üí lowest slot, drawn first; nearest paints on top) using preallocated scratch (zero per-frame allocation, GC-neutral); [Game](src/Game.ts) calls it once per frame at the end of `onFrame`, after positions settle, before the render. Depth key = `dot(pos, cameraLookDir)` (planar/along-view, the correct painter's key); a stable sort + an "already sorted? skip the repack+upload" guard keep static boards free. Cost is negligible (one dot + an O(n log n) index sort + a 13-float/instance gather at tens-to-low-hundreds of sprites). **Verified:** 5 headless [SpriteRenderer.test.ts](src/render/SpriteRenderer.test.ts) cases (far‚Üínear order, handle‚áÑdata integrity through the permutation, idempotence, removal-then-sort, <2 no-op) + a live browser integration check on `dev-preview` ‚Äî driving the real `onFrame` closure by hand (the preview tab throttles rAF to ~0, the documented gotcha) sorted a live 50-sprite battle buffer to **0 out-of-order transitions every frame across 500 frames** of combat, no console errors. The SUBJECTIVE occlusion look (does the melee no longer pop in front of the closer target) still wants the user's native-browser playtest. Render-only ‚Äî no `src/sim`, no snapshot/fuzz change.

- [x] **#3 ‚Äî A kiting archer paths ON TOP of an enemy melee when trapped in a corridor (a real same-cell overlap).** ‚úÖ **FIXED (Qb, 2026-06-18).** Root cause (confirmed by a headless trace + a RED corridor-pin test, [MovementBehavior.test.ts](src/sim/behaviors/MovementBehavior.test.ts) "corridor kite-pin (Qb #3)"): a ranged unit one cell INSIDE `minRange` can neither fire (the band gates firing too ‚Äî [strikes.ts](src/sim/abilities/strikes.ts):85) nor hold, so it wants to KITE out; but in a 1-wide corridor the walls kill the E5.B perpendicular sidestep and a friendly melee fills the only back-step. With the kite blocked, `MovementBehavior`'s goal list fell through to its LAST goal ‚Äî the **target's own cell** (the APPROACH anti-freeze fallback) ‚Äî and since the target is **soft-EXCLUDED** from the step-collision set (`excludeUnitId`, so `findPath` always has a reachable goal + the in-range abstain stops an *approaching* unit a cell short), `advance` stepped the archer straight ONTO the hostile's cell. The in-range abstain that normally stops it short never fired because the unit was too CLOSE (inside the band), not too far. **Fix** ([MovementBehavior.ts](src/sim/behaviors/MovementBehavior.ts)): only append the target-cell fallback goal when `dist >= minRange` (the approach / in-band-no-LOS cases); when KITING (`dist < minRange`) omit it, so a pinned kiter abstains/queues instead of charging onto the soft-excluded target. Byte-identical for melee (`minRange 0`) + every too-far approach; the only behavior change is the blocked-kiter case. **Verified:** the new corridor invariant test (no two combatants share a cell at a tick boundary) + 1016 main + typecheck + **205 fuzz:smoke byte-identical** (no re-baseline ‚Äî the case doesn't arise in the smoke seeds). *(The original investigation notes are preserved in git history.)*

## Design explorations (post-Phase-G)

- [x] **Movement abilities (dash / gap-closer) ‚Äî the real rogue fix.** ‚úÖ **DONE (N1, 2026-06-14):** the rogue dash shipped ‚Äî an aggressive-close gap-closer (range 2 / 0.25s motion / 10s cd), the ability config promoted to a `kind` discriminated union (`attack|heal|movement`, runtime stays flat), a first-class `unit:dashed` event ([DashAction](src/sim/actions/DashAction.ts)). **Still N2's:** flip the rogue to `weakest` targeting + re-run this forced-roster eval now that the backline is reachable. The original disproof that motivated mobility is kept: H7c added a
      general per-archetype **targeting strategy** field ([config/archetypes.json](config/archetypes.json) +
      [src/sim/targetingStrategies.ts](src/sim/targetingStrategies.ts): `nearest` / `weakest`). The plan was to
      make the rogue target the **weakest** (squishiest) enemy = "backline assassin." A
      forced-roster eval **disproved it for our rogue**: `weakest` *halved* the rogue's
      damage (6.1 ‚Üí 3.1 dmg/dep) and the free search still never recruited it. Mechanism:
      the rogue is **range 1** with no gap-closer, so committing to the farthest squishy
      mark makes it walk *past* adjacent enemies it could strike (the strike only fires on
      the committed target), dying en route. `weakest` is left **registered but unassigned**
      (all archetypes = `nearest`) ‚Äî it's the right strategy for a unit that can actually
      *reach* the backline. So the real rogue fix is **mobility**: a dash / leap / blink
      ability (a new ability + likely a small action-phase + movement-intent seam) that
      closes distance to the assassination target, at which point flipping the rogue to
      `weakest` becomes viable and measurable (re-run the same forced-roster eval). Pairs
      with the broader deferred rogue identity work (evasion / stealing ‚Äî Phase I+ / shop).
      Surfaced 2026-06-07 during the H7c rogue pass; full data in [BALANCE.md](BALANCE.md).
