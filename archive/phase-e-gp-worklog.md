- **E5 complete** — pathfinding refresh (target stickiness + boids sidestep) + the per-ability range migration folded in. Five commits:
  1. **Range migration** (`0c7099c`). Attack range left the archetype (`config/archetypes.json#attackRange` is gone) for `config/abilities.json` alongside `cooldownSeconds`, so E7's multi-ability units can carry a melee swing and a long-range bolt with different reaches. `proposeBasicStrike` gates on the *firing ability's* own range; `derived.attackRange` is redefined as the unit's **effective engagement range = max over its abilities** (computed via the renamed `rangeForArchetype`, threaded through `deriveStats` unchanged), which MovementBehavior's in-range abstain + the HUD/audio readers consult exactly as before. Behavior-preserving for today's single-ability archetypes (each unit's ability range == its old archetype range) → no determinism/fuzz shift, **no snapshot bump**. The lighter-touch version of the migration (kept `derived.attackRange` as a cached max rather than deleting it — avoids a third snapshot bump and keeps the single display/audio value).
  2. **E5.A — target stickiness** (`84ecde0`). New `Unit.targetId` + `outOfLosTicks` (snapshotted → **WorldSnapshot v12→v13**; v12 throws). `updateTarget(unit, world)` runs once per *free* unit in the selector (before behaviors poll, so the re-target decision + LOS counter advance exactly once/tick, not once per behavior). Re-target rules: (a) committed target died/vanished → nearest; (b) a rival is ≥ `SIM.retargetCloserRatio` (1.5×) closer → switch; (c) ranged only: target out of LOS for `SIM.rangedRetargetLosTicks` (0.5s) → drop + re-pick. MovementBehavior + the strike abilities read the commitment via `currentTarget` (sticky id, else nearest fallback so a direct `proposeAction` call without a prior `updateTarget` still works). Knobs in new `config/sim.json` (A4, authored in seconds). `collectLosBlockers` hoisted from strikes.ts to Targeting.ts. **This carries most of the fix** — verified live: all 5 player units converged on one `targetId` instead of thrashing.
  3. **E5.B — boids sidestep + tie-break** (`95db0e0`). When MovementBehavior's A*-chosen `path[1]` is occupied, it tries a one-cell perpendicular sidestep toward the target (2 candidates per the decision point) before abstaining; abstains only when neither perpendicular is free (1-wide corridor → queue still emerges). **The sidestep is a marginal safety net**: soft-block A* (gotcha #38's +100 cost) already routes around an ally whenever there's lateral room (path[1] not occupied → no sidestep needed) and queues when there isn't (no lateral cell to sidestep to anyway), so it only fires in forced chokepoints. Kept per ROADMAP + harmless + forward-looking for E7's denser formations; easy to drop if you want leaner code. Also: straight-line A* tie-break in `popLowestF` (f-ties break toward the goal, then lex key) replacing the insertion-order tie-break that crabbed paths leftward — pure equal-f ordering, so admissibility (gotcha #34) holds and paths stay min-cost; only *which* min-cost path changes.
  4. **Fuzz cap fix** (`ffdb8c8`). `DEFAULT_MAX_TICKS` was a hardcoded `1000` with a stale "≈100s at TICK_RATE=10" comment — E3.5's TICK_RATE 10→20 silently halved it to ~50s, so grown-team battles on the 32-long endlessCorridors board read as false **hangs** (a 6v5 battle hitting exactly 1000 ticks with 4 deaths each side = progressing, not deadlocked). Now `secondsToTicks(100)` so it tracks TICK_RATE (gotcha #6). Caught while re-running the post-E5 fuzz baseline; NOT an E5 deadlock.
  5. **Soft-block penalty → tunable knob, 100→4** (`ca9e794`). Follow-up after playtest: the user still saw *all* units backtracking *very early*. Root cause was the `OCCUPIED_CELL_PENALTY = 100` hardcode (gotcha #38) — so steep A\* would take a ~99-cell detour rather than step through a single ally, so units on the packed spawn band peeled into long backward flanks before engaging. Hoisted to `config/sim.json#occupiedCellPenalty`; a 24-seed endlessCorridors sweep put the knee at **4** (early-backward steps 6.2%→3.4%, avg battle ~911→~707 ticks, 0 hangs at every value since the penalty stays finite). At 4, A\* only detours around an ally when it costs ≤3 extra cells, else routes through → the step-collision queue / E5.B sidestep (so the sidestep is **no longer near-dormant** — see gotcha #104/#107). Only affects ally/non-target-enemy cells; walls + half-cover are hard blockers, untouched. **User-playtested as "much improved."** Tune by feel in `config/sim.json`.

  - **[SupportMovementBehavior](src/sim/behaviors/SupportMovementBehavior.ts) yield rule:** when the healer would otherwise idle/hold AND it sits on the **only** cell a living ally can advance through (`blockedAlly`), it proposes a `SwapAction` with that ally (score 1, so a ready `heal_ally` at 10 still wins — the swap only fires on heal-cooldown ticks). Two non-obvious pieces, both found by tracing:
    - **"Forward" is grid PATH distance, not Chebyshev.** In a funnel the route to an enemy runs *away* from it in straight-line terms (back through a gap, then around), so a Chebyshev "is this cell forward" test never fires. `blockedAlly` runs a **BFS distance field from the ally's `currentTarget`** (over the static neutral-wall topology, cached per target) — `h` blocks the ally iff `dist[h] < dist[ally]` and no *other* passable+unoccupied neighbour of the ally is closer. New helpers `distanceField` / `neutralCells` / `isNavigable` (the last two also seed GP5.2's snap). Occupancy test is **neutral-INCLUSIVE** (half-cover is a neutral *unit*, not a tile, so `tileGrid.costAt` reads finite — `passable` only rejects it via the occupied set).
    - **The yield must fire from the BLOCKED branches, not just the idle tail.** The healer is wedged exactly when an enemy is in panic range (step 2) or it's trying to trail a centroid the other side of a 1-wide row (step 4) — those branches `return`ed before the idle-tail yield. So steps 2/3/4 now fall through to `yieldChokepoint` when their own move is blocked. **A multi-healer regression was caught + fixed here:** with the *forward-vacate* version, three healers funnelling deadlocked the whole army at the spawn-row gap (a healer wanting to trail east vs. a fighter wanting the gap west — opposite directions in the 1-wide y=0 row, step 4 returned null and never swapped). The **swap** (sends the support to the rear, never leading) + the step-4 fall-through drains it.
